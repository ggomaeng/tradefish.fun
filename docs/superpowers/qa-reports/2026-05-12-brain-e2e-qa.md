## Bugs Found

### BUG-1 (HIGH): Migration 0011_brain.sql uses `uuid_generate_v4()` ‚Äî not available on remote Supabase

**Severity:** HIGH ‚Äî blocks all brain migrations from applying.

**Repro:**  
```
supabase db push --include-all
```
Fails with: `ERROR: function uuid_generate_v4() does not exist (SQLSTATE 42883)`

**Root cause:** `agent_retrievals` table in 0011_brain.sql line 90 used `uuid_generate_v4()`, but the remote Supabase project does not have `uuid-ossp` enabled. All newer migrations in the codebase use `gen_random_uuid()` (built-in PostgreSQL 13+).

**Fix applied** (inline, <5 lines):  
`supabase/migrations/0011_brain.sql` line 90: `uuid_generate_v4()` ‚Üí `gen_random_uuid()`

**Verified:** Migration pushes successfully after fix.

---

### BUG-2 (HIGH): Cosmograph graph fails to render ‚Äî missing required `pointIndexBy` prop

**Severity:** HIGH ‚Äî graph canvas shows error "Failed to get points data. Missing required properties: pointIndexBy" and the force-directed visualization does not render.

**Repro:**  
1. Navigate to `/brain` with at least 1 `wiki_entries` row  
2. Cosmograph logs: `üö´ [Cosmograph > validatePoints]: Failed to get points data. Missing required properties: "pointIndexBy".`

**Root cause:** `BrainGraph.tsx` passed `pointIdBy="id"` to `<Cosmograph>` but did not pass `pointIndexBy`. Cosmograph's `validatePoints()` requires both: `pointIdBy` (string ID column) and `pointIndexBy` (integer index column). The integer column is used for efficient GPU-side lookups.

**Fix applied** (inline, <5 lines):  
`src/components/brain/BrainGraph.tsx`:
- Added `_index: i` to each point record in `toPointRecords()`
- Added `pointIndexBy="_index"` prop to `<CosmographComponent>`

**Verified:** Graph renders correctly after fix. Node visible in canvas.

---

### BUG-3 (LOW): Scrubber causes React hydration mismatch

**Severity:** LOW ‚Äî cosmetic console error, no functional impact.

**Repro:**  
Browser console shows hydration mismatch on Scrubber thumb/fill styles (numeric `top: -4` vs string `top: "-4px"`) on every page load.

**Root cause:** Scrubber uses numeric inline style values (e.g. `top: -4`) in `thumbStyle` and related constants. React SSR serializes these as pixel strings, but the client renders them as numbers, causing a diff.

**Fix:** Change `thumbStyle` numeric values to explicit pixel strings (`top: "-4px"`, `width: "12px"`, etc.) OR add `suppressHydrationWarning` to the Scrubber container. Low severity ‚Äî not fixing in this QA pass; flagged for follow-up.

---

### BUG-4 (LOW): Realtime `lessons today` counter double-counts on initial node insert

**Severity:** LOW ‚Äî cosmetic display issue.

**Repro:**  
1. Load `/brain` with 1 node (lessons=1 shown)
2. Insert a new `wiki_entries` row via Realtime
3. Counter jumps to 3 instead of 2

**Root cause:** On realtime insert, `handleNodeInsert` calls `incrementCounter("lesson")` (+1 delta). The SidePanel's `computeInitialCounters` also re-runs on the next render since `data.nodes` changed (now includes the new node). So `initialCounters.lesson` = 2, plus delta = 1, gives display value = 3.

**Fix:** `initialCounters` should be memoized on mount only (use an initial snapshot, not the live `data.nodes`), or the delta should only track events after mount. Not fixing inline ‚Äî medium-effort refactor of `useBrainCounters`. Flag for fullstack-engineer.
